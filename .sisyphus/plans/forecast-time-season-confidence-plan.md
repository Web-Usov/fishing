## 1) Цель и границы

Реализовать улучшение качества и правдоподобия прогноза клёва за счёт:

1. добавления фактора времени суток;
2. добавления сезонного фактора;
3. пересмотра формулы confidence;
4. сохранения explainability в коротком читаемом формате;
5. обновления тестов и документации.

При этом **не менять публичный API** (схемы `shared-zod`, сигнатуры сервиса и контроллера).

## 4) Детальный план реализации

### Шаг 1. Базовый анализ текущей реализации

1. Зафиксировать текущие функции скоринга и текущие пороги `level/confidence`.
2. Подтвердить, что `timestamp` и `timezone` приходят в доменную функцию и не используются.
3. Зафиксировать текущий порядок формирования `factors` и выбор `strongestFactor`.

**Критерий готовности шага:** есть точная baseline-картина перед изменениями.

### Шаг 2. Внедрить фактор времени суток

1. Добавить функцию:

```ts
function scoreTimeOfDay(timestamp: string, timezone: string): BiteForecastFactor
```

2. Реализовать получение локального часа из `timestamp + timezone` (детерминированно, без внешних API).
3. Применить мягкую шкалу импактов (без резких скачков), где:
   - 04:00–08:59: выраженный плюс;
   - 18:00–22:59: заметный плюс;
   - 11:00–16:59: небольшой минус;
   - 23:00–03:59: выраженный минус;
   - переходные окна: нейтрально/слегка.
4. Подобрать импакты в рамках текущей модели (ориентир: не сильнее основных погодных факторов ~10–14).
5. Добавить фактор в массив `factors` рядом с остальными.
6. Проверить человекочитаемые label’ы для explainability (например, «Утренний выход рыбы», «Ночная пассивность рыбы»).

**Критерий готовности шага:** при изменении времени/таймзоны меняется `timeOfDay` factor и может меняться итоговый score.

### Шаг 3. Внедрить фактор сезонности

1. Добавить функцию:

```ts
function scoreSeason(timestamp: string): BiteForecastFactor
```

2. Определять месяц из `timestamp`, классифицировать сезон (северное полушарие):
   - зима: 12/1/2;
   - весна: 3/4/5;
   - лето: 6/7/8;
   - осень: 9/10/11.
3. Реализовать soft-bias импакты:
   - весна/осень: небольшой плюс;
   - лето: нейтрально или лёгкий плюс;
   - зима: нейтрально или лёгкий минус.
4. Держать модуль импактов в вспомогательном диапазоне (до 6–8).
5. Добавить фактор в `factors` после `scoreTimeOfDay`.

**Критерий готовности шага:** смена сезона даёт ожидаемый небольшой сдвиг прогноза.

### Шаг 4. Пересчитать confidence

1. Заменить текущую формулу на:

```ts
const offset = Math.abs(score - 50)
if (offset >= 25) return 'high'
if (offset >= 15) return 'medium'
return 'low'
```

2. Убедиться, что `high` становится редким и возникает только далеко от середины.
3. Проверить, что около `score ~ 50` преобладает `low`.

**Критерий готовности шага:** confidence соответствует новой логике и лучше отражает «уверенность простой модели».

### Шаг 5. Explainability и strongestFactor

1. Убедиться, что новые факторы участвуют в выборе `strongestFactor` на равных.
2. Проверить, что label новых факторов естественно читается в фразе объяснения.
3. Сохранить один короткий sentence в explanation (формат не ломать).
4. Опционально: добавить `(+X/-Y)` к ключевому фактору только если это не ухудшает UX.

**Критерий готовности шага:** объяснение короткое, понятное, и в релевантных кейсах отражает время суток/сезон.

### Шаг 6. Тесты

Обязательные сценарии unit-тестов в `domain-bite-forecast`:

1. **Раннее утро осенью + хорошая погода**:
   - `score >= 60`;
   - `level` минимум `good`;
   - `strongestFactor.label` указывает на суточный или сезонный контекст (один из ожидаемых label’ов новых факторов).

2. **Полдень летом + средняя погода**:
   - `score` в окне `40..60`;
   - `confidence !== 'high'`.

3. **Глубокая ночь зимой + плохая погода**:
   - `score <= 39`;
   - `level === 'poor'`;
   - `confidence` соответствует offset-логике (`high` только при `abs(score - 50) >= 25`).

Дополнительно:

- точечные тесты границ временных диапазонов (03:59/04:00/08:59/09:00 и т.д.);
- тесты сезонных границ (февраль→март, май→июнь, август→сентябрь, ноябрь→декабрь);
- тест инварианта диапазонов ответа (`score`, `impact`, enum-поля).

**Критерий готовности шага:** тесты подтверждают новую логику и инварианты контракта.

## 5) Риски и контроль

1. **Риск:** некорректная обработка timezone/локального часа.
   - **Контроль:** тесты на граничные часы и валидные IANA timezone.

2. **Риск:** новые факторы «перетягивают» модель.
   - **Контроль:** ограничение импактов и проверка распределения score на регрессионных наборах.

3. **Риск:** drift между кодом и docs.
   - **Контроль:** правка docs в том же цикле + `pnpm docs:build`.

4. **Риск:** случайное изменение публичного контракта.
   - **Контроль:** явный no-change check для `shared-zod`/API сигнатур.

## 6) Definition of Done

Готово, когда одновременно выполнено всё:

1. API-контракт и сигнатуры не изменены.
2. В коде `domain-bite-forecast` добавлены `scoreTimeOfDay` и `scoreSeason`, оба участвуют в итоговом прогнозе.
3. `confidence` работает по `offset`-формуле.
4. Explainability остаётся короткой и в релевантных кейсах отражает новые факторы.
5. Unit-тесты покрывают ключевые сценарии (утро-осень, полдень-лето, ночь-зима + границы).
6. Обновлены `bite-forecast-model.md`, `domain-model.md`, `delivery/task-state.md` (и при необходимости `product-overview.md`/`mvp-scope.md`).
7. `pnpm test` и `pnpm docs:build` проходят успешно.
