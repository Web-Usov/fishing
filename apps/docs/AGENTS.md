# AGENTS: работа с документацией (`apps/docs`)

Этот файл описывает правила для агентов и разработчиков, которые обновляют документацию проекта.

## 1. Источник истины

- Каноническая документация проекта находится в `apps/docs/docs/*`.
- Любые изменения в коде, влияющие на поведение системы, должны быть отражены в этой папке в том же цикле работ.

## 2. Что обновлять при изменениях

- `docs/mvp-scope.md` — при изменении пользовательского функционала и границ MVP.
- `docs/architecture.md` — при изменении архитектуры, рантайма, интеграций, портов, деплоя, LAN-ограничений.
- `docs/shared-code-plan.md` — при изменении границ переиспользования между web/mobile/shared пакетами.
- `docs/index.md` — при изменении структуры разделов или правил ведения документации.

## 3. Правила оформления

- Основной язык документации — русский.
- Заголовки должны быть конкретными, без общих формулировок.
- Технические термины и имена файлов/пакетов оставлять в оригинале (`apps/web`, `packages/shared-ui` и т.д.).
- Не дублировать устаревшую информацию: при изменении поведения обновлять существующие разделы, а не добавлять противоречащие заметки.

## 4. Проверка перед завершением

После правок обязательно выполнить:

```bash
pnpm docs:build
```

Если изменения затрагивают monorepo-процессы или интеграции, дополнительно:

```bash
pnpm typecheck
pnpm build
pnpm test
```

## 5. Локальный запуск документации

```bash
pnpm docs:install
pnpm docs:serve
```

Docs-сервис в Docker Compose:

```bash
docker compose up -d --build docs
```

Открыть: `http://localhost:8000`
