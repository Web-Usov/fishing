# Backlog

Backlog нужен для декомпозиции продукта на крупные user stories и технические блоки, которые можно раздавать в разные агент-сессии.

## Правило декомпозиции

Каждый блок должен иметь:

1. продуктовый смысл;
2. разбиение по сервисам/платформам;
3. понятные границы ownership.

## User Story: Прогноз клёва

Как рыболов, я хочу выбрать точку и получить прогноз клёва, чтобы решить, стоит ли ехать на рыбалку.

### Техническая декомпозиция

- API: расчёт, контракт, runtime, тесты.
- Web: карта, UI прогноза, explainability, состояния.
- Mobile: перенос пользовательского сценария.
- Admin: будущие инструменты контроля качества прогнозного блока.
- Shared: доменный движок, weather normalization, схемы.

## User Story: Журнал улова

Как рыболов, я хочу сохранять результат рыбалки, чтобы сравнивать прогноз с фактом и накапливать свою историю.

### Техническая декомпозиция

- API: CRUD и модель данных.
- Web: формы и история.
- Mobile: быстрый field-first сценарий записи.
- Admin: операционный просмотр и контроль данных.
- Shared: схемы и доменные сущности.

## User Story: История точек и избранное

Как пользователь, я хочу возвращаться к важным точкам, чтобы не начинать сценарий каждый раз заново.

### Техническая декомпозиция

- хранение точек;
- история использования;
- быстрый повтор сценариев на web/mobile.

## User Story: Социальный слой

Как рыболов, я хочу видеть отчёты других пользователей, чтобы опираться не только на модель, но и на живой контекст.

### Техническая декомпозиция

- API: отчёты, контент, модерация.
- Web/Mobile: лента и просмотр контента.
- Admin: moderation workflows.

## Technical Block: Platform maturity

- CI/CD
- quality gates
- инфраструктура
- безопасность
- docs-driven planning и agent workflow

## Как использовать backlog в агентной работе

- одна сессия берёт один блок или одну вложенную подзадачу;
- если задача многосервисная, ownership делится по сервисам;
- перед началом работы агент обязан свериться с `task-state.md`.
