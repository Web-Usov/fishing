# Архитектура

Этот документ описывает общую техническую картину без глубокого погружения в детали каждого сервиса. Детализация вынесена в отдельные сервисные документы.

Документ — основной вход в **Develop-режим**: с него начинается техническое чтение системы перед изменениями в коде и инфраструктуре.

## Архитектурная идея

Проект строится как монорепозиторий, в котором пользовательские поверхности, серверные части, инфраструктура и документация развиваются как отдельные сервисные зоны, а общие контракты, доменные модели и технические примитивы выносятся в shared packages.

Базовый принцип: **переиспользовать смысл, а не смешивать платформы**. Домен, контракты и часть инфраструктуры должны быть общими. UX, навигация и платформенные интеграции должны оставаться на уровне приложений.

## Основные сервисы и зоны ответственности

### Пользовательские поверхности

- `apps/web` — основной текущий MVP-клиент с картой и прогнозом клёва.
- `apps/mobile` — мобильная оболочка и будущий field-first интерфейс рыболова.
- `apps/admin` — внутренняя административная поверхность.
- `apps/docs` — документационный сервис и источник истины по продукту и разработке.

### Серверный слой

- `apps/api` — HTTP API на NestJS. Сейчас основной доменный контур — прогноз клёва, в будущем сюда должны заходить остальные серверные модули.

### Инфраструктурный слой

- `postgres` — основное хранилище будущих persisted-данных.
- `packages/db` — Prisma-слой схемы и доступа к данным.

### Shared-слой

- `packages/domain-bite-forecast` — расчёт прогноза клёва.
- `packages/domain-geo` — геодоменные примитивы.
- `packages/shared-zod` — общие контракты и схемы данных.
- `packages/api-client` — клиент к API и нормализация погодных данных.
- `packages/shared-ui` / `packages/ui` — UI-примитивы и reusable-слой.
- `packages/config-*` — общие конфигурации инструментов.

## Как части системы взаимодействуют

Текущий основной runtime-поток:

1. Пользователь работает с картой в `apps/web`.
2. Web получает погодный контекст через same-origin endpoint `/api/weather/forecast`.
3. Web вызывает `apps/api` для расчёта прогноза клёва.
4. API использует shared-контракты и доменный движок.
5. Результат возвращается в web и отображается как 7-дневный прогноз.

В будущем этот же смысловой поток должен быть повторён в mobile, а admin должен закрывать внутренние сценарии управления системой.

## Стек верхнего уровня

- Monorepo: `pnpm workspace` + `turbo`
- Web/Admin: `Next.js`
- Mobile: `Expo` / `React Native`
- API: `NestJS`
- Validation/contracts: `Zod`
- Database: `Postgres` + `Prisma`
- Docs: `Docusaurus`
- Tests: `Vitest` + API smoke/e2e
- Контейнеризация: `Docker Compose`

## Развёртывание и LAN

- Основной локальный orchestration-контур — Docker Compose из корня репозитория.
- Команды:
  - `pnpm docker:up`
  - `pnpm docker:rebuild`
  - `pnpm docker:dev`
- Типовые LAN-точки входа:
  - web: `http://<host-ip>:3010`
  - admin: `http://<host-ip>:3012`
  - api: `http://<host-ip>:3001`
  - expo: `http://<host-ip>:8081`
- `docker-compose.yaml` использует `develop.watch` для live sync сервисов.

## Web runtime и hydration

- Первый рендер должен быть детерминированным.
- Browser-only источники (`window`, `localStorage`, `navigator`, `location`) читаются только после mount.
- URL-состояние (`lat/lng`) синхронизируется только после первичного чтения query.
- Runtime info, locale и theme строятся через hydration-safe подход.

## Актуальные архитектурные примечания

- API отдаёт CORS-заголовки для LAN MVP-доступа.
- Web определяет API host по текущему browser host для локального сетевого сценария.
- Реальная погода берётся из Open-Meteo через same-origin proxy route.
- Если внешний погодный источник недоступен, web переходит в fallback-режим и явно помечает пониженную точность.
- Поле `moonIlluminationPct` пока получает нейтральный fallback.
- Карта строится через provider adapter pattern (`yandex` / `google`).

## Архитектурные принципы

- Один монорепозиторий — единая инженерная картина продукта.
- Shared domain и shared contracts — до платформенной реализации.
- Карта/координата — первичный пользовательский вход в сценарий.
- Документация — часть архитектуры управления проектом, а не побочный артефакт.
- Любой крупный блок должен отражаться одновременно в продуктовых, архитектурных и delivery-документах.
