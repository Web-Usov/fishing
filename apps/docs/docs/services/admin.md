# Admin

## Назначение

`apps/admin` — административная поверхность продукта. Сейчас это минимальная оболочка, которая резервирует отдельный контур под внутренние и backoffice-сценарии.

## Доменный смысл и зона ответственности

Admin отвечает за внутренние процессы качества и эксплуатации платформы, а не за публичный пользовательский сценарий.

## Входы и выходы

- управление доменными справочниками;
- контроль качества данных;
- внутренние операционные задачи;
- будущая модерация пользовательского контента.

Основные взаимодействия:

- **Входы:** внутренние операционные действия команды и администраторов.
- **Выходы:** управленческие изменения в данных, модерация и операционные состояния системы.

## Текущее состояние

- есть отдельное приложение в монорепозитории;
- есть отдельный runtime и собственный UI-слой;
- полноценные административные workflows ещё не реализованы.

## Стек

- `Next.js`
- `React`
- `TypeScript`
- `@fishing/ui`

## Зависимости

- `apps/api` (административные серверные сценарии);
- `@fishing/ui` и внутренний UI-слой;
- будущие persisted-данные через Postgres/Prisma.

## Принципы

- admin не должен быть копией публичного web;
- это отдельный внутренний контур со своей информационной архитектурой;
- все будущие административные возможности должны проходить через отдельную документацию и backlog-декомпозицию.

## Инфраструктура и качество

- сервис `admin` в Docker Compose;
- обязательные build/typecheck;
- в будущем — отдельные CRUD/test сценарии и операционные runbooks.

## Типовые сценарии изменений

- Добавить admin workflow: уточнить ownership в backlog, затем реализовать API+UI связку.
- Изменить справочник/модерацию: синхронизировать контракты, server runtime и интерфейс админа.
- Добавить операционный экран: описать сценарий и ограничения в docs до расширения функционала.
