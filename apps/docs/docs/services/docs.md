# Docs

## Назначение

`apps/docs` — документационный сервис проекта. Это не просто сайт с markdown-файлами, а управленческий контур, в котором живут описание продукта, архитектура, roadmap, backlog и правила для агентной разработки.

## Доменный смысл и зона ответственности

Docs отвечает за согласованную картину продукта и разработки: здесь фиксируются канонические правила, границы MVP, архитектурные решения и состояние delivery-потоков.

## Входы и выходы

Docs выполняет четыре функции:

- source of truth;
- навигация по знаниям о проекте;
- инструмент декомпозиции разработки;
- точка синхронизации между параллельными агент-сессиями.

Основные взаимодействия:

- **Входы:** изменения в коде, архитектуре, продуктовых сценариях и delivery-процессе.
- **Выходы:** актуализированные документы в `apps/docs/docs/*`, валидная навигация и единые правила работы.

## Стек

- `Docusaurus`
- ручная sidebar-навигация
- markdown-first workflow

## Зависимости

- root `AGENTS.md` и `apps/docs/AGENTS.md`;
- `sidebars.js` и `docusaurus.config.js`;
- обязательная сборка docs как quality gate.

## Правила качества

- любые существенные изменения в коде требуют обновления docs;
- `pnpm docs:build` обязателен;
- broken links не допускаются;
- root `AGENTS.md` и `apps/docs/AGENTS.md` задают правила работы с документацией.

## Почему docs выделены в отдельный сервис

Потому что проекту нужен не набор случайных заметок, а структурированная knowledge system, которая помогает параллельно развивать продукт и не терять целостную картину.

## Типовые сценарии изменений

- Изменился продуктовый сценарий: обновить `product-overview.md`/`mvp-scope.md` и связанные delivery-документы.
- Изменились архитектура или сервисные границы: обновить `architecture.md`, `services/*` и при необходимости `shared-code-plan.md`.
- Изменилась навигация: синхронно обновить `index.md` и `sidebars.js`.
