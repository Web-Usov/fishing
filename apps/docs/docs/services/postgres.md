# Postgres

## Назначение

`postgres` — инфраструктурный сервис хранения данных. Сейчас он является foundation-слоем для будущих persisted сценариев и уже присутствует в runtime-контуре проекта.

## Доменный смысл и зона ответственности

Postgres нужен как основа для:

- пользовательских данных;
- журнала улова;
- справочников и доменных сущностей;
- административных и аналитических сценариев.

## Входы и выходы

- **Входы:** операции чтения/записи через централизованный Prisma-слой в `packages/db`.
- **Выходы:** persisted-данные для API, admin-контуров и будущих продуктовых модулей.
- Прямой доступ из приложений должен оставаться исключением; стандарт — через серверный слой и контракты.

## Текущее состояние

- сервис поднимается через Docker Compose;
- используется volume для хранения данных;
- readiness контролируется через healthcheck;
- Prisma-слой находится в `packages/db`.

## Зависимости

- Docker Compose runtime;
- `packages/db` как единая точка схемы и доступа;
- `apps/api` и будущие server-side модули, которые используют хранилище.

## Архитектурные принципы

- схема данных должна управляться централизованно;
- доступ к данным не должен расползаться напрямую по приложениям;
- любые изменения хранения должны отражаться в коде и документации одновременно.

## Направление развития

- полноценная продуктовая модель данных;
- миграции и runbooks;
- эксплуатационные практики backup/recovery по мере роста продукта.

## Типовые сценарии изменений

- Добавить новую сущность: обновить Prisma-схему, миграции и связанный сервисный контур API.
- Изменить модель хранения: синхронно обновить код, docs и delivery-координацию.
- Подготовить operational практику: фиксировать runbooks и ограничения в service docs.
