# Postgres

## Назначение

`postgres` — основное хранилище persisted-данных продукта. На него опираются текущая и будущие серверные модели данных (журнал улова, пользовательские данные, настройки и другие состояния, которые должны переживать перезапуск приложения).

## Расположение и стек

- инфраструктурный сервис `postgres` описан в `docker-compose.yaml`;
- Prisma-клиент и схема данных находятся в `packages/db` (`@fishing/db`);
- источник подключения — `DATABASE_URL` с postgres-dsn в runtime API-контуре.

## Связь с API и доменом

- API-контур использует Postgres только через слой `packages/db` (единая точка схемы и доступа к данным).
- Доменные пакеты (например, `@fishing/domain-bite-forecast`) не должны зависеть от таблиц напрямую: они работают с абстрактными входами/выходами и бизнес-правилами.
- На текущем этапе модуль прогноза клёва работает без persisted-записи; подключение Postgres уже есть в runtime-инфраструктуре и готово для следующих серверных модулей.

## Миграции: текущий фактический статус

- Prisma-схема расположена в `packages/db/prisma/schema.prisma`.
- Папка миграций Prisma (`packages/db/prisma/migrations`) в текущем состоянии репозитория отсутствует.
- В `packages/db/package.json` есть Prisma-команда генерации клиента: `prisma generate`.
- Единого workspace-скрипта для наката миграций (через `package.json`) пока не заведено — это необходимо добавить при переходе к активным изменениям схемы.

Базовое правило: при изменении схемы должны создаваться и применяться миграции через единый стандартизованный workflow (после добавления соответствующего скрипта в monorepo).

## Будущие доменные модули

Ближайшие продуктовые области, которые будут активно использовать БД:

- журнал улова;
- пользовательские профили и предпочтения;
- история точек и избранное;
- административные и служебные контуры данных.

## Текущее состояние runtime

- сервис `postgres` запускается в Docker Compose с healthcheck и отдельным volume для данных;
- API-сервис в том же compose-контуре зависит от `postgres` и получает `DATABASE_URL` на этот сервис.

## Архитектурные принципы

- централизованная схема и доступ к данным через `packages/db`;
- отсутствие прямых SQL/ORM-зависимостей в доменных пакетах;
- любое изменение схемы должно синхронно отражаться в коде, документации и delivery-планировании.
