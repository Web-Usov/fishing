# Описание продукта

Этот документ — вход в **Product-режим** документации. Он фиксирует продуктовый смысл, целевые сегменты и ценность без технических деталей реализации.

## Что это за продукт

Fishing — это цифровое приложение для рыболовов, которое помогает понять, **куда ехать, когда ехать и чего ожидать от конкретной точки**. Продукт строится вокруг решений пользователя до, во время и после рыбалки.

Это не просто погодный интерфейс и не просто карта. Это система принятия решений для рыболова, которая со временем должна соединить прогноз, геоконтекст, личную историю улова и коллективный слой пользовательских данных.

## Для кого продукт

### 1. «Редко выезжающий»

Рыболов, который ездит нерегулярно и не хочет тратить редкий выезд на точку с низкой вероятностью результата.

### 2. «Регулярный»

Рыболов, который выезжает часто, сравнивает несколько привычных точек и хочет быстрее выбирать рабочий вариант на ближайшие дни.

### 3. «Спортивный/продвинутый»

Пользователь, для которого важны системность, накопление наблюдений, сравнение прогноза с фактом и личная точность решений по точкам.

### 4. «Оператор/админ»

Роль платформы, которая поддерживает качество данных и контентный контур, чтобы пользовательские сценарии оставались понятными и полезными.

## Какую проблему решает продукт

Сегодня у рыболова информация часто разорвана:

- погода — в одном месте;
- карта — в другом;
- личный опыт — только в памяти;
- коллективный опыт — в чатах, форумах и случайных источниках.

Fishing должен собрать это в один рабочий пользовательский цикл.

## Бизнесовая ценность

Продукт создаёт ценность на трёх уровнях:

### 1. Короткий горизонт: польза здесь и сейчас

Пользователь быстро отвечает себе на вопрос: «Есть ли смысл ехать в эту точку сейчас или в ближайшие дни?»

### 2. Средний горизонт: персональная история

Приложение должно постепенно становиться личным журналом рыболова, а не одноразовой прогнозной витриной.

### 3. Дальний горизонт: платформенный и социальный эффект

После накопления базы сценариев и данных продукт может расти в сторону community, рекомендаций, пользовательских отчётов и сервисной платформы вокруг рыбалки.

## Продуктовый сценарий верхнего уровня

### Сегодня

1. Открыть карту.
2. Выбрать точку.
3. Получить прогноз клёва.
4. Посмотреть детализацию.
5. Принять решение о выезде.

### Целевой расширенный сценарий

1. Выбрать точку или водоём.
2. Получить прогноз, контекст и объяснение.
3. Зафиксировать выезд и улов.
4. Сравнить прогноз с фактом.
5. Использовать персональную и коллективную историю для следующих выездов.

## Сценарии использования по сегментам

### «Редко выезжающий»

**Сейчас в MVP:**

1. Открывает карту перед выходными.
2. Выбирает конкретную точку на знакомом водоёме.
3. Смотрит 7-дневный прогноз и факторы по выбранной дате.
4. Принимает решение, ехать в эту точку или перенести выезд.

**Запланировано дальше:**

1. Фиксирует факт выезда и результат в журнале.
2. Сравнивает, совпал ли прогноз с реальным результатом.
3. Накапливает личную историю по своим редким, но важным поездкам.
4. Использует этот контекст при следующем выборе точки.

### «Регулярный»

**Сейчас в MVP:**

1. Открывает карту в начале недели.
2. Проверяет несколько точек по очереди через coordinate-first сценарий.
3. Сравнивает прогноз и объяснение факторов на ближайшие дни.
4. Выбирает точку и время следующего выезда.

**Запланировано дальше:**

1. Ведёт историю выездов по каждой рабочей точке.
2. Видит связь между условиями и фактическим уловом в своих записях.
3. Получает более персональный контекст для регулярного планирования.
4. Использует будущий social-слой как дополнительный сигнал, а не замену личному опыту.

### «Спортивный/продвинутый»

**Сейчас в MVP:**

1. Выбирает точку на карте под конкретную сессию.
2. Анализирует прогноз и детализацию факторов перед выездом.
3. Формирует рабочую гипотезу по времени и условиям ловли.
4. Принимает решение о тактике на выезд.

**Запланировано дальше:**

1. Фиксирует сессию и улов в структурированном журнале.
2. Сопоставляет просмотренный прогноз с фактическим результатом.
3. Отслеживает личную точность модели по своим точкам.
4. Использует персональную историю и будущие коллективные сигналы для более точной стратегии.

### «Оператор/админ»

**Сейчас в MVP:**

1. Контролирует, что базовый контур «карта → точка → прогноз» остаётся стабильным.
2. Следит за ясностью объяснений факторов и пользовательских состояний.
3. Поддерживает единый продуктовый контекст в документации и контенте.

**Запланировано дальше:**

1. Поддерживает консистентность слоя истории выездов и журнала улова.
2. Следит, чтобы персональные и будущие social-сигналы не подменяли базовую прозрачность прогноза.
3. Поддерживает качество данных для сравнения «прогноз против факта».
4. Обеспечивает понятную эволюцию продукта без размывания границы MVP.

## Ключевые продуктовые направления

### Прогноз клёва

Первый уже реализуемый и проверяемый слой продукта.

### Карта и геоконтекст

Основной вход в сценарий. Пользователь мыслит местом и точкой, а не страницей приложения.

### Журнал улова

Следующий необходимый слой после прогноза. Подробно описан в документе [Журнал улова и калибровка модели](./catch-journal.md).

Смысл этого слоя в том, чтобы перевести личный опыт из памяти в проверяемую историю: пользователь фиксирует выезд и результат, затем сравнивает факт с тем прогнозом, который видел до выезда. Такая связка формирует персональный контур доверия к прогнозу и даёт основу для оценки качества модели по реальным данным пользователя.

Важно: полноценный журнал улова относится к следующему этапу развития и не входит в текущий MVP.

### Социальный контур

Отчёты, рекомендации, посты, активности на точках и коллективный рыболовный контекст.

## Продуктовые принципы

- Сначала практическая польза, потом сложность.
- Сначала работающий сценарий, потом расширение функционала.
- Любая модель должна быть объяснима человеку.
- Карта — это первичный продуктовый объект, а не декоративный элемент.
- Web и mobile должны быть согласованы по смыслу, но не обязаны быть одинаковыми по UX.

## Текущий этап

Сейчас продукт находится на стадии раннего map-first MVP. Прогноз клёва — это первая реальная форма взаимодействия, вокруг которой строится и проверяется вся следующая продуктовая система.
