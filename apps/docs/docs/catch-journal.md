# Журнал улова и калибровка модели

Этот документ описывает следующий крупный доменный слой после MVP baseline прогноза клёва. Речь идёт о целевой функциональности, которая будет добавлена после текущего map-first этапа, а не о уже доступной части продукта.

См. связанные документы: [Объём MVP](mvp-scope.md), [Описание продукта](product-overview.md), [Доменная модель](domain-model.md), [Модель прогноза клёва](bite-forecast-model.md).

## Роль журнала улова

Журнал улова нужен, чтобы перевести пользовательский цикл из режима «посмотрел прогноз и поехал» в режим накопления проверяемого опыта. Пользователь фиксирует выезд и результат не в памяти, а в структуре данных, которую потом можно сравнивать с прогнозом.

Базовая запись будущего журнала включает:

- дату и время выезда;
- водоём и точку ловли;
- прогноз, который пользователь видел перед выездом;
- фактический результат по улову.

Эта связка формирует прозрачную цепочку «прогноз → условия → результат», которая нужна и пользователю, и продукту.

## Связь с доменной моделью

Журнал опирается на уже зафиксированные сущности `Trip`, `Catch`, `Forecast`, `Spot`, `Waterbody`.

- `Trip` хранит факт или план выезда, время и контекст.
- `Spot` и `Waterbody` задают географический уровень, точка и водоём.
- `Forecast` фиксирует прогноз, на который пользователь опирался до выезда.
- `Catch` хранит фактические результаты в рамках конкретного выезда.

На уровне связей это даёт единый контур: пользователь выбирает `Spot` или `Waterbody`, смотрит `Forecast`, проводит `Trip`, фиксирует `Catch`, после чего можно сравнивать ожидание и факт по одной и той же ситуации.

## Калибровка/оценка модели

Текущая модель прогноза в MVP остаётся экспертной и ручной, она не является self-learning системой. Журнал улова нужен как будущая основа для оценки качества и аккуратной калибровки, а не как мгновенный автообучающий контур.

Первый слой будущих метрик:

- качество прогноза на уровне конкретного пользователя;
- качество прогноза по конкретному `Waterbody`;
- качество прогноза по региону;
- стабильность качества по сезонам и типам условий.

Отдельный пользовательский сигнал доверия, который должен появиться на основе журнала, это понятная оценка «насколько модель хороша в его точках». Такой сигнал помогает принимать решение о выезде на основе личного опыта, а не только общей шкалы прогноза.

## Ограничения и приоритеты

Важно явно зафиксировать границу: полноценный журнал улова не реализован в текущем MVP, см. [Объём MVP](mvp-scope.md). В текущей поставляемой версии есть только baseline прогноза для выбранной координаты и объяснение факторов.

Приоритет следующего этапа после MVP baseline:

1. добавить структурированную фиксацию `Trip` и `Catch`;
2. связать фактические записи с просмотренным `Forecast`;
3. запустить прозрачную оценку качества без заявлений о полном автоматическом обучении модели.

Это позволяет развивать систему по шагам, сначала проверяемая история и доверие пользователя, затем более глубокая калибровка и персонализация.
