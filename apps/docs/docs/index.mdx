# Документация Fishing

`apps/docs/docs/*` — источник истины по текущему состоянию продукта, архитектуры и плана развития.

## Как пользоваться документацией

- Сначала выберите режим чтения: [**Product**](product-start.md) или [**Develop**](develop-start.md).
- Каноническая документация проекта находится в `apps/docs/docs/*`; если есть расхождение с устными договорённостями, приоритет у docs.
- Не создавайте дублирующую правду: при изменениях обновляйте релевантные документы в этой структуре.
- Если меняется код, продуктовый сценарий, архитектура или delivery-процесс, это должно быть отражено в docs в той же work cycle.
- Для активной работы и ownership всегда сверяйтесь с `delivery/plans/index.md` и конкретным `PLAN.md` потока.

## Полная карта документов

Эта документация отвечает на пять практических вопросов:

1. Что мы строим как продукт и для кого.
2. Как устроена система на уровне сервисов, инфраструктуры и связей.
3. Как устроен каждый сервис отдельно.
4. Как декомпозирована разработка и какие планы можно брать в параллельные агент-сессии.
5. Какие документы агент обязан обновлять вместе с кодом.

## Принцип ведения документации

- Любое изменение в коде, которое меняет продукт, архитектуру, сервисные границы или delivery-процесс, должно быть отражено в этой папке.
- Root `AGENTS.md` и `apps/docs/AGENTS.md` описывают, как агент должен работать с этой структурой.
- Если устное описание и документация расходятся, приоритет у `apps/docs/docs/*`.

## Быстрые команды

- `pnpm docs:serve` — локальный запуск документации
- `pnpm docs:build` — обязательная проверка после правок
- `pnpm docker:dev` — контейнерный dev-контур с watch-синхронизацией
- `pnpm docker:rebuild` — полная пересборка контейнеров
