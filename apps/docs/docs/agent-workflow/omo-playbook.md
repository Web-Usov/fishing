# OmO Agent Workflow Playbook

Практический playbook для команды Fishing по продуктивной работе в Oh-My-OpenCode (OmO): каких агентов и навыки выбирать, когда включать определённые режимы, и как не тратить время на неправильный orchestration.

## Для чего этот документ

- Ускорить delivery без потери качества.
- Снизить число "дорогих" или лишних запусков агентов.
- Стандартизировать подход между сессиями и уменьшить конфликт при параллельной работе.

## Базовый принцип выбора: **домен → агент/категория → навык**

1. Сначала определяем домен задачи (UI, архитектура, docs, отладка, исследование кода и т.д.).
2. Затем выбираем **агента или категорию**, оптимизированную под этот домен.
3. Только после этого подключаем релевантные навыки (skills), если они реально усиливают задачу.

Если сомневаетесь между несколькими вариантами — выбирайте более узкоспециализированный путь, а не универсальный.

## Быстрый выбор (decision table)

| Ситуация | Что выбрать | Когда эскалировать |
|---|---|---|
| Локальная правка до ~1 часа, понятный scope | `quick` + основной исполнитель | Если после 2 итераций нет прогресса → `deep` |
| UI/UX, визуальные состояния, browser сценарии | `visual-engineering` + `playwright` | Если нужен системный редизайн и спорные trade-off’ы → `deep`/`oracle` |
| Кросс-модульная задача (`apps/*` + `packages/*`) | `deep` (+ `metis` при высокой неоднозначности) | Если остаются конкурирующие архитектурные варианты → `oracle` |
| Тяжёлая логика или нетривиальный алгоритм | `ultrabrain` | Если риск затрагивает security/perf/архитектуру целиком → `oracle` |

## Default flow для 80% задач

1. Выберите `quick` или `deep` по сложности и масштабу.
2. Назначьте 1 основной агент под домен задачи.
3. Подключайте skill только по явной пользе (не "на всякий случай").
4. Если после 2 целевых итераций прогресса нет — эскалируйте (`deep`/`ultrabrain`/`oracle`).
5. Закройте задачу верификацией и синхронизацией документации.

## Карта агентов OmO для нашего монорепо

### `explore` (codebase discovery)

Использовать, когда нужно быстро понять существующий код:

- где реализован конкретный сценарий в `apps/api`, `apps/web`, `apps/mobile`;
- где уже есть похожие паттерны в `packages/*`;
- какие файлы затрагивает изменение перед реализацией.

**Не использовать** для тривиальных задач (одна известная строка в известном файле).

### `librarian` (external references)

Использовать, когда вопрос касается внешних библиотек, best practices, OSS-примеров:

- интеграция нового пакета в `apps/api`;
- выбор подхода для SDK/клиентской библиотеки в `packages/api-client`;
- сравнение внешних паттернов перед архитектурным решением.

**Не использовать** для поиска по нашему репозиторию (для этого `explore`).

### `oracle` (high-IQ read-only consultation)

Использовать точечно, когда цена ошибки высока:

- архитектурный trade-off между несколькими сервисами;
- сложный баг после 2+ неудачных попыток;
- риски безопасности/производительности;
- итоговый review значимой реализации.

**Важно:** Oracle — дорогой ресурс. Это консультант, а не исполнитель.

#### Жёсткие правила вызова Oracle

Используем `oracle` только если выполнен хотя бы один триггер:

- блокер сохраняется после 2 самостоятельных и осмысленных попыток;
- high-risk решение по security, performance или архитектуре;
- цена ошибки высока (критичный пользовательский поток или кросс-сервисный контракт).

**Не используем `oracle`** для:

- рутинных CRUD-правок;
- форматирования и мелких doc-правок;
- стандартных рефакторингов без архитектурного риска.

### `metis` и `momus`

- `metis` — pre-planning анализ: уточняет скрытые неоднозначности до старта сложной работы.
- `momus` — строгая критика и валидация плана/результата перед масштабным внедрением.

Используйте их там, где план влияет на много модулей (`apps/*` + `packages/*`) или на delivery-процесс.

## Категории `task(...)`: когда какая

### `quick`

Только для действительно мелких правок:

- 1 файл;
- понятный scope;
- минимум логики.

### `visual-engineering`

Любые изменения UI/UX и визуального поведения:

- компоненты в `apps/web`, `apps/admin`, `apps/mobile`;
- layout, spacing, states, анимации;
- дизайн-консистентность.

### `deep`

Для полноценных end-to-end задач с исследованием и реализацией:

- кросс-модульные изменения;
- сложные интеграции и миграции;
- задачи, где важно сначала глубоко изучить контекст.

### `ultrabrain`

Для действительно трудных задач с высокой когнитивной сложностью:

- нетривиальные архитектурные развилки;
- algorithmic/logic-heavy решения.

### `writing`

Для системной документации и больших текстовых изменений:

- крупные playbook/гайд-документы;
- унификация формулировок и структур.

## Навыки (skills): как подключать осмысленно

Подключайте skill только когда он реально улучшает результат задачи.

- `frontend-ui-ux` — для UI/UX задач и визуальной доработки интерфейсов.
- `playwright` / `dev-browser` — когда нужно браузерное взаимодействие, проверка сценариев, скриншоты.
- `git-master` — любые нетривиальные git-операции (история, атомарные коммиты, анализ изменений).
- `review-work` — post-implementation review значимых изменений.
- `find-skills` — когда нужно подобрать новый skill под конкретную задачу.
- `searxng-search` — расширенный web/package поиск для внешнего исследования.

### Обязательные соответствия (must use)

- Любые browser-задачи (навигация, скриншоты, веб-валидация) → обязательно `playwright`.
- Нетривиальные git-операции (history search, commit strategy, rebase/squash) → обязательно `git-master`.
- Пост-имплементационная проверка значимых изменений → `review-work`.

## Режимы и паттерны работы в OmO

### 1) Background research first

Для нетривиальных задач сначала запускаем исследовательские агенты в фоне (`explore`/`librarian`), затем параллельно делаем не пересекающуюся подготовку.

### 2) Consultation before irreversible design

Если решение влияет на архитектурные границы, перед реализацией делаем консультацию (`oracle`), а не "впихиваем" решение сразу в код.

### 3) Verify after every meaningful change

После изменения — обязательная верификация (для docs минимум `pnpm docs:build`, для кода дополнительно релевантные typecheck/build/test).

### 4) One intent, one flow

Не смешивать в одном потоке независимые цели. Для каждой большой цели — отдельный `PLAN.md`, отдельный execution-flow и отдельный `REVIEW.md`.

### 5) Docs first, plan second, code third

Перед реализацией значимого изменения порядок должен быть таким:

1. подтвердить или обновить product / architecture docs;
2. зафиксировать поток в `delivery/plans/<plan>/PLAN.md`;
3. только после этого идти в код;
4. завершать поток через `REVIEW.md`.

## Готовые рецепты для Fishing

### Рецепт A: "Нужно понять, где менять бизнес-логику прогноза"

1. `explore` по `packages/domain-bite-forecast` + местам использования в `apps/api`.
2. Проверка контрактов в `packages/shared-zod` и `packages/api-client`.
3. Реализация изменений.
4. Верификация и синхронизация docs.

### Рецепт B: "Новая UI-фича в web/admin"

1. Категория `visual-engineering` (+ `frontend-ui-ux`).
2. Переиспользование shared-компонентов (`packages/shared-ui`, `packages/ui`).
3. Проверка UX-состояний, затем тест/сборка.

### Рецепт C: "Сложная деградация производительности API"

1. Локализуем узкое место базовыми измерениями.
2. Если остаются конкурирующие гипотезы — консультация `oracle`.
3. Фикс с минимальным риском регрессии.
4. Проверка результата и фиксация в docs при изменении архитектурных соглашений.

### Рецепт D: "Новый внешний пакет и неочевидный integration path"

1. `librarian` для официальных практик и примеров OSS.
2. Сверка с текущими ограничениями архитектуры (`architecture.md`, service docs).
3. Реализация + верификация + docs sync.

## Антипаттерны (избегать)

- Использовать `oracle` на рутине (дорого, не ускоряет).
- Делать UI-задачи вне `visual-engineering`.
- Дублировать поиск: сначала делегировали `explore`, потом вручную ищем то же самое.
- Начинать код без сверки с каноническими docs в `apps/docs/docs/*`.
- Закрывать задачу без верификации и без обновления документации при изменении поведения.

### Антипаттерн → как правильно

- **Неправильно:** звать `oracle` при любой неопределённости.
  **Правильно:** сначала 2 целевые итерации в выбранной категории, затем точечная эскалация по триггерам.
- **Неправильно:** подключать много skills "для подстраховки".
  **Правильно:** 1–2 навыка, которые напрямую соответствуют типу задачи.
- **Неправильно:** смешивать независимые цели в одной сессии.
  **Правильно:** один логический поток на одну цель, с отдельным `PLAN.md` и `REVIEW.md`.

## Чеклист перед стартом любой значимой задачи

- [ ] Определён тип задачи: Product / Develop / смешанная.
- [ ] Прочитаны входные документы (`product-start.md` и/или `develop-start.md`).
- [ ] Проверен `delivery/plans/index.md` на существующие и пересекающиеся потоки.
- [ ] Понятно, какой `PLAN.md` создаётся или обновляется в этом work cycle.
- [ ] Выбран правильный агент/категория под домен.
- [ ] Подключены только релевантные skills.
- [ ] Понятен план верификации (docs:build / typecheck / build / test).
- [ ] Понятно, какие документы обновляются в этом work cycle.

## Где этот playbook использовать в первую очередь

- При онбординге новых участников в OmO-процессы.
- Перед кросс-сервисными изменениями (`apps/*` + `packages/*`).
- При запуске параллельных агент-сессий, чтобы снизить конфликт и дублирование работы.
